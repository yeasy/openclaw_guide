## 16.3 OpenAI 与本地模型集成

OpenClaw 不绑定单一 AI 供应商。除 Claude 外，OpenAI 与以 Ollama 为代表的本地模型也是生产环境中常见的接入选择。本节介绍这两类生态的接入方式和典型配置，以及与 Claude 组合使用的混合策略。

### 16.3.1 OpenAI 模型接入

OpenAI 官方模型 ID（如 `gpt-5.5`、`gpt-5.4`）不等于 OpenClaw 中的 provider-qualified model ref。直接 OpenAI Platform API 路由通常写成 `openai/<model>`，例如 `openai/gpt-5.5`；ChatGPT/Codex 订阅 OAuth 路由写成 `openai-codex/<model>`，例如 `openai-codex/gpt-5.4`。两条路由的认证来源、限额与可用模型可能不同。当前 OpenAI 最新旗舰模型为 GPT-5.5（Standard short-context $5/$30 per M tokens），前代 GPT-5.4（$2.50/$15 per M tokens）及其 mini/nano 变体仍可使用；是否能在本机 OpenClaw 中调用，还要用 `openclaw models list --provider openai` / `openclaw models list --provider openai-codex` 核对 catalog，并用 `openclaw models status --probe` 实测认证与 entitlement。

#### 基础配置

```jsonc
{
  models: {
    providers: {
      openai: {
        apiKey: "${OPENAI_API_KEY}",
      },
    },
  },
  agents: {
    defaults: {
      model: {
        primary: "openai/gpt-5.4",
        fallbacks: ["openai/gpt-5.5"],
      },
    },
  },
}
```

#### 使用 Azure OpenAI

企业用户通常通过 Azure OpenAI Service 调用 GPT 模型。当前 OpenClaw 文档已经把 Azure 的 Responses 形态单独区分为 Azure provider 路径，实际 endpoint、deployment 与 provider 标识请以当前 Azure provider 文档为准；不要简单把 API Key 模式下的 `openai/*` 片段原样照抄到 Azure 环境。

```jsonc
{
  models: {
    providers: {
      azureOpenai: {
        apiKey: "${AZURE_OPENAI_API_KEY}",
        baseUrl: "https://your-resource.openai.azure.com/openai/deployments/your-deployment",
        apiVersion: "2024-02-01",
      },
    },
  },
}
```

#### OpenAI 接入要点

| 能力 | 说明 |
|------|------|
| **工具调用** | Function Calling 与 OpenClaw 工具系统兼容，格式自动转换 |
| **结构化输出** | `response_format: { type: "json_schema" }` 可配合 OpenClaw 输出解析 |
| **o 系列推理模型** | `openai/o3` 等推理模型仍可使用 system message，但对提示层级更敏感；建议把稳定角色约束放在 system message，把任务细节留在 user message |
| **Embeddings** | OpenAI Embeddings API 可通过 MCP 服务器或自定义工具接入 |

> **注意**：OpenAI 模型版本与定价以 [OpenAI 官方文档](https://platform.openai.com/docs/models) 为准。本书示例中的模型标识符仅反映撰写时的版本。

### 16.3.2 本地模型：Ollama 接入

Ollama 是主流的本地模型运行平台，支持 Llama、Mistral、Qwen、DeepSeek 等开源模型。本地部署的优势是数据不出本地网络、无 API 调用费用，适合对隐私敏感或网络受限的场景。

#### 接入原理

OpenClaw 当前推荐通过 **Ollama 原生 API** 接入本地模型。不要把 OpenClaw 指向 `http://localhost:11434/v1` 这样的 OpenAI-compatible 路径；官方明确说明这会破坏工具调用，让模型把工具 JSON 当普通文本输出。正确做法是直接使用 Ollama 的原生 `baseUrl`：

```jsonc
{
  models: {
    providers: {
      ollama: {
        baseUrl: "http://localhost:11434",
      },
    },
  },
  agents: {
    defaults: {
      model: {
        primary: "ollama/mistral-small:22b",
      },
    },
  },
}
```

#### 启动 Ollama 并拉取模型

```bash
# 安装 Ollama（macOS）
brew install ollama

# 启动服务
ollama serve

# 拉取模型（以 Mistral Small 22B 为例）
ollama pull mistral-small:22b

# 验证服务
curl http://localhost:11434/api/tags
```

#### 本地模型选型参考

| 模型 | 参数量 | 适用场景 | 显存需求 |
|------|--------|----------|----------|
| `llama3.1:8b` | 8B 级 | 快速问答、通用助手 | ~5-6GB |
| `llama3.3:70b` | 70B 级 | 更强通用对话、多语种任务 | ~43GB 模型文件，实际显存看量化与运行参数 |
| `mistral-small:22b` | 22B | 通用推理、工具调用 | ~14GB |
| `qwen2.5-coder:32b` | 32B | 代码生成、代码审查 | ~20GB |
| `deepseek-r1:32b` | 32B | 复杂推理、代码生成 | ~20GB |

> 实际显存占用取决于量化精度（Q4/Q8 等），以 `ollama show <model>` 输出为准。

### 16.3.3 多生态混合策略

在实际生产中，不同任务对模型能力和成本的要求不同，混合使用多生态模型可以兼顾质量与效率。

#### 典型混合配置

```jsonc
{
  agents: {
    // 通用对话：用本地模型降低成本
    chat: {
      model: {
        primary: "ollama/mistral-small:22b",
        fallbacks: ["openai/gpt-5.4"],
      },
    },
    // 复杂推理与代码生成：用 Claude Sonnet
    dev_assistant: {
      model: {
        primary: "anthropic/claude-sonnet-4-6",
        fallbacks: ["openai/gpt-5.5"],
      },
    },
    // 高并发轻量任务：用 OpenAI 小模型
    classifier: {
      model: {
        primary: "openai/gpt-5.4",
      },
    },
  },
}
```

#### 混合策略选型参考

| 策略 | 主模型 | Fallback | 适用场景 |
|------|--------|----------|----------|
| 隐私优先 | 本地（Ollama） | `anthropic/claude-haiku-4-5-20251001` | 敏感数据不出本地 |
| 成本优先 | 本地（Ollama） | `openai/gpt-5.4` | 高并发、低复杂度 |
| 质量优先 | `anthropic/claude-sonnet-4-6` | `openai/gpt-5.5` | 推理质量要求高 |
| 离线兜底 | `anthropic/claude-sonnet-4-6` | 本地（Ollama） | 网络不稳定环境 |

Fallback 链路的触发条件和行为详见 [4.4 模型降级与 Fallback 策略](../04_config_models/4.4_failover.md)。

### 16.3.4 统一管理多生态模型

无论接入哪家供应商，OpenClaw 的模型管理界面提供统一视图。可通过以下命令查看当前所有已配置模型的状态：

```bash
openclaw models status --probe
```

建议在混合部署中：
1. 为每个供应商配置独立的 `apiKey` 环境变量，避免密钥混用。
2. 对本地模型定期运行探针，确认 Ollama 服务存活。
3. 将各供应商的配额告警阈值统一纳入监控，防止某一路径静默失效。
