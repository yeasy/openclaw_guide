# 第七章 多渠道分发与多智能体协作

前几章，你已经把 OpenClaw 装在了本地，学会了在 Dashboard 对话框里聊天。现在，你要把它扩展到真实的生产环境——让它在 Telegram、WhatsApp、飞书企业群里工作，或者分拆成多个专业智能体各司其职。

当入口从单渠道扩展到多渠道、多群与多对端后，系统最常见的问题不是模型能力不足，而是边界不清：谁负责处理、哪些入口允许触发、哪些能力属于高风险、出现问题如何回放定位。这就是为什么本章高度关注“治理”而不只是“集成”。

本章的主线是把多渠道与多智能体做成可管理系统：

- 渠道策略负责收敛触发面（哪些人、哪些群可以触发）。
- 路由绑定负责收敛所有权（哪个智能体接管哪个入口）。
- 工具策略与沙箱约束负责收敛可执行边界（什么操作允许做）。

**学完这章你能做什么**

- 自信地把 OpenClaw 接入 Telegram、WhatsApp 或飞书，而不只是“跟着文档复制粘贴”
- 设计出符合自己业务的渠道隔离策略（比如客服和内部运维分开走不同账号）
- 设置多个智能体，每个专注于不同的领域或用户群，通过路由框架自动分发请求
