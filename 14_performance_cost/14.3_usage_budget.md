## 14.3 OpenClaw 的用量观测与外部预算治理

本节介绍如何通过 OpenClaw 内置命令、Dashboard 的 `Control -> Usage` 页面建立**实时用量观测**，再结合 provider billing limit、外部监控、入口限流或自定义插件实现预算治理。OpenClaw 当前主要提供证据面；硬封顶、告警通知和自动限流通常属于外部控制面。

## 14.3.1 核心观测命令

OpenClaw 提供多个内置命令用于实时观测与诊断。若你更习惯图形化视图，当前 Dashboard 也提供 `Control -> Usage` 页面，可按时间范围查看 Token、成本与会话分布；如果你已经在聊天 / auto-reply 命令路径里通过 `/usage cost` 发现异常，再回到 Usage 页做系统级排查通常更高效。注意 Dashboard Chat 的本地 `/usage` 是会话 token / context 视图，不等同于聊天侧 `/usage cost` 成本摘要。

### `/status` 命令

查看当前会话或系统的实时状态。

```bash
/status
```

输出示例：
```yaml
OpenClaw Gateway Status (build: 2026.03.15)

会话统计：
  活跃会话: 42
  Token 使用 (今日): 245.6K / 500K (49%)

成本（今日）:
  LLM: $7.35
  工具: $2.10
  存储: $0.05
  总计: $9.50 / $50.00 预算 (19%)
```

#### `/usage cost` 命令

聊天 / auto-reply 命令路径内的本地成本摘要，适合快速看最近一段时间的费用分布。若需要 CLI 视角的完整使用量信息，可配合 `openclaw status --usage`；若需要图形化视图，可对照 Dashboard 的 `Control -> Usage` 页面。

```bash
/usage cost
```

输出：
```text
成本分析（最近 24 小时）

按 Agent 统计：
  support_agent: $23.45 (65%)
  data_analyst_agent: $10.20 (28%)
  routing_agent: $2.15 (6%)
  其他: $0.20 (1%)

按模型统计：
  Claude Sonnet 4.6: $28.50 (79%)
  Claude Haiku 4.5: $7.50 (21%)

按操作统计：
  LLM 推理: $28.00 (78%)
  API 调用: $6.50 (18%)
  存储: $1.50 (4%)
```

#### `/compact` 命令

手动触发上下文压缩，回收 Token 预算。

```bash
/compact 聚焦保留 API 设计决策
```

输出：
```text
压缩完成：
  原始 Token 数: 12350
  压缩后 Token 数: 4820 (61% 回收)
```

### 14.3.2 预算配置与告警

#### 全局预算限制

> [!CAUTION]
> 下列内容更适合作为**治理目标**而不是“当前 OpenClaw 已内置的预算 schema”。截至当前版本，OpenClaw 主要提供 usage / cost 可见性与 provider quota 快照；预算封顶、告警通知、异常检测通常需要外部监控或插件补齐。

推荐落地方式：

- 用 `/status`、`/usage full` 和 `openclaw gateway usage-cost --json` 获取会话与近期开销快照。
- 用 provider 自带 quota / billing 控制台设硬预算。
- 用 Prometheus、Grafana、Loki、外部 Webhook 或自定义插件做阈值告警。

#### 按 Agent 预算分配

如果你确实需要“按 Agent 配额”，应把它实现为外部治理策略，例如：

- 为不同 Agent 绑定不同 provider / auth-profile / model 层级。
- 在消息入口、调度层或插件层做额度判断。
- 定期导出 usage-cost 数据，按 agentId 做聚合与报表。

### 14.3.3 上下文管理机制

#### 自动压缩与清理

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "anthropic/claude-haiku-4-5"
      }
    }
  }
}
```

当前 compaction 配置挂在 `agents.defaults.compaction`；会话保留与清理由 `session.maintenance` 负责，而不是根级 `context.expiration` 或 `agents.list[].compaction`。

#### 会话重置与清理

```bash
# 在当前聊天里重置上下文
/reset

# 预览会话清理
openclaw sessions cleanup --dry-run
# （使用 --dry-run 预览要删除的内容，不实际删除）

# 实际执行清理
openclaw sessions cleanup --enforce
```

### 14.3.4 成本控制的工程策略

#### 策略 1：分层模型选择

根据任务复杂度做**工程分层**是合理的，但当前官方配置面并没有内置“自动复杂度检测 + 路由 rules”这套 schema。更可执行的做法是：

- 为不同 agent 直接指定不同默认模型。
- 用 `fallbacks` 做可靠性兜底，而不是把成本路由逻辑塞进未文档化字段。
- 若要按复杂度动态切换，放到自定义插件、调度器或上层业务入口中实现。

#### 策略 2：工具缓存与重复消除

对重复的工具调用做缓存本身是合理优化，但它通常属于**具体插件或上游系统**的实现细节，而不是当前 OpenClaw 核心公开的 `tools.caching` 配置面。写文档时应明确这是工程建议，不是现成 schema。

### 14.3.5 成本异常检测与预警

通过日志、会话用量与外部监控实现异常消费告警是推荐做法，但基线建模、统计阈值、自动暂停等逻辑应放在监控系统、自动化平台或自定义插件中实现，而不是假设 OpenClaw 已内置 `anomalyDetection` schema。

### 14.3.6 用量观测与外部预算治理检查清单

部署 OpenClaw 时，确保完成以下配置：

- **可见性**：启用 `/usage full`、定期导出 `openclaw gateway usage-cost --json`
- **预算控制**：在 provider 控制台或外部监控里设硬阈值
- **自动压缩**：根据实际模型窗口配置 `agents.defaults.compaction`
- **会话清理**：使用 `openclaw sessions cleanup --dry-run/--enforce` 并配置 `session.maintenance`
- **成本分析**：每周检查 `openclaw status --usage` 与 usage-cost 报表
- **模型分层**：按 agent / 入口拆分模型，而不是依赖未文档化的自动复杂度路由

完成以上配置，可实现**可观、可控、可优化**的成本管理体系。
