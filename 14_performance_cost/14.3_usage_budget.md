## 14.3 OpenClaw 的用量观测与预算控制

本节介绍如何通过 OpenClaw 内置命令与配置实现**实时用量监控**与**精细化预算控制**，防止成本失控。

### 14.3.1 核心观测命令

OpenClaw 提供多个内置命令用于实时观测与诊断。

#### `/status` 命令

查看当前会话或系统的实时状态。

```bash
/status
```

输出示例：
```

会话统计：
  活跃会话: 42
  Token 使用 (今日): 245.6K / 500K (49%)

模型调用：
  Total: 523 calls
  Success: 502 (96%)
  Throttled: 15 (3%)
  Failed: 6 (1%)

工具执行：
  成功: 1203
  失败: 12
  平均延迟: 234ms

成本（今日）:
  LLM: $7.35
  工具: $2.10
  存储: $0.05
  总计: $9.50 / $50.00 预算 (19%)
```

#### `/cost` 命令

详细的成本分析，按不同维度聚合。

```bash
/cost --period=24h --breakdown=agent
```

输出：
```
成本分析（最近 24 小时）

按 Agent 统计：
  support_agent: $23.45 (65%)
  data_analyst_agent: $10.20 (28%)
  routing_agent: $2.15 (6%)

按操作统计：
  LLM 推理: $28.00 (78%)
  API 调用: $6.50 (18%)
  存储: $1.50 (4%)
```

#### `/compact` 命令

手动触发上下文压缩，回收 Token 预算。

```bash
/compact --session=<session_id> --target-tokens=5000
```

输出：
```

  原始 Token 数: 12350
  压缩后 Token 数: 4820 (61% 回收)
  方法: 消息合并 + 优先级裁剪
  质量评分: 0.98 (压缩对会话理解的影响)
```

### 14.3.2 预算配置与告警

#### 全局预算限制

在 `openclaw.json` 中配置全局预算：

```json
{
  "budget": {
    "daily": {
      "tokens": 1000000,
      "cost": 500,
      "requests": 100000
    },
    "monthly": {
      "tokens": 30000000,
      "cost": 15000
    }
  },
  "alerts": {
    "enabled": true,
    "thresholds": {
      "dailyTokens": 0.8,
      "dailyCost": 0.9,
      "monthlyTokens": 0.75
    },
    "channels": ["email", "slack"]
  }
}
```

**配置解读**：
- 日预算：100 万 Token 或 $500（先达到的限制）
- 月预算：3000 万 Token 或 $15,000
- 告警触发条件：日 Token 达 80%、日成本达 90%、月 Token 达 75%
- 告警渠道：邮件 + Slack 通知

#### 按 Agent 预算分配

```json
{
  "agents": {
    "support_agent": {
      "budget": {
        "dailyTokens": 500000,
        "dailyCost": 250,
        "enforceLimit": true
      }
    },
    "research_agent": {
      "budget": {
        "dailyTokens": 200000,
        "dailyCost": 100,
        "enforceLimit": false
      }
    }
  }
}
```

**`enforceLimit` 选项**：
- `true`：超预算后新请求被拒绝，返回"预算已用尽"错误
- `false`：超预算后发出告警但继续处理（用于非关键 Agent）

### 14.3.3 上下文管理机制

#### 自动压缩与清理

配置自动压缩触发器：

```json
{
  "context": {
    "compaction": {
      "enabled": true,
      "triggerThreshold": 8000,
      "targetTokens": 5000,
      "preserveRecentMessages": 5,
      "summarizationModel": "gpt-3.5-turbo",
      "schedule": "on-demand"
    },
    "expiration": {
      "maxAge": "7d",
      "idleAge": "24h",
      "action": "archive"
    }
  }
}
```

**参数说明**：
- `triggerThreshold: 8000`：当上下文超 8K Token 时自动压缩
- `targetTokens: 5000`：压缩目标为 5K Token
- `summarizationModel: "gpt-3.5-turbo"`：用便宜的模型做摘要
- `maxAge: "7d"`：会话超过 7 天自动归档
- `idleAge: "24h"`：24 小时未活动的会话进入空闲状态（减少存储）

#### 会话重置与清理

```bash
# 主动重置某个会话（清空上下文，开始新对话）
/reset --session=<session_id>

# 批量清理过期会话
openclaw cleanup --older-than=30d --dry-run
# （使用 --dry-run 预览要删除的内容，不实际删除）

# 实际执行清理
openclaw cleanup --older-than=30d
```

### 14.3.4 成本控制的工程策略

#### 策略 1：分层模型选择

根据任务复杂度自动选择模型，降低平均成本：

```json
{
  "routing": {
    "complexity_detection": {
      "enabled": true,
      "rules": [
        {
          "condition": { "complexity": "simple" },
          "model": "claude-3.5-haiku",
          "criteria": "查询字数 < 50 且不涉及推理"
        },
        {
          "condition": { "complexity": "medium" },
          "model": "claude-3.5-sonnet",
          "criteria": "一般问题"
        },
        {
          "condition": { "complexity": "high" },
          "model": "claude-3-opus",
          "criteria": "多步推理、代码编写、深度分析"
        }
      ]
    }
  }
}
```

**效果**：
- 简单问题（占 70%）用 Haiku 成本降低 80%
- 中等问题（占 25%）用 Sonnet 平衡质量与成本
- 复杂问题（占 5%）用 Opus 保证质量

**预期成本降幅**：55% 左右。

#### 策略 2：工具缓存与重复消除

对重复的工具调用进行缓存：

```json
{
  "tools": {
    "caching": {
      "enabled": true,
      "ttl": 3600,
      "keyStrategy": "hash(tool, args)",
      "excludeTools": ["send_email", "create_ticket"]
    }
  }
}
```

**原理**：
- 用户 A 查询"2026 年苹果股价" → 调用 API（花费 $0.005）
- 30 秒后，用户 B 查询同一问题 → 返回缓存结果（0 成本）
- 24 小时内缓存命中率 30-50%

**注意**：某些工具不能缓存（如发邮件），在 `excludeTools` 中列出。

#### 策略 3：离峰处理与异步任务

对非实时任务进行异步处理，避免高峰时段成本：

```json
{
  "scheduling": {
    "asyncTasks": {
      "enabled": true,
      "offPeakHours": "22:00-06:00",
      "priorityTiers": {
        "realtime": 1,
        "normal": 2,
        "batch": 3
      }
    }
  }
}
```

**实例**：
- 实时：客服回复（优先级 1）
- 普通：数据分析请求（优先级 2）
- 批处理：日志清理、报表生成（优先级 3，在夜间处理）

### 14.3.5 成本异常检测与预警

OpenClaw 可以检测异常消费模式并自动告警：

```json
{
  "anomalyDetection": {
    "enabled": true,
    "baselines": {
      "averageDailyCost": 500,
      "stdDeviation": 100
    },
    "triggers": [
      {
        "name": "spike_detection",
        "condition": "dailyCost > baseline + 3 * stdDev",
        "action": "alert + pause_non_critical"
      },
      {
        "name": "model_drift",
        "condition": "error_rate > 5%",
        "action": "notify_admin"
      }
    ]
  }
}
```

**实际应用**：
- 如果今日成本突然从 $500 跳到 $1200，自动暂停非关键 Agent，防止更大损失
- 如果某个模型错误率突增到 5%，自动触发故障回退

### 14.3.6 用量观测与预算控制检查清单

部署 OpenClaw 时，确保完成以下配置：

1. **日预算设置** ✓
   - [ ] 全局日成本预算（建议为历史平均 + 20%）
   - [ ] 按 Agent 分配预算

2. **告警配置** ✓
   - [ ] 启用告警通知
   - [ ] 配置告警渠道（Slack、邮件等）
   - [ ] 设置合理告警阈值（如 80% 触发告警）

3. **自动压缩** ✓
   - [ ] 启用 `context.compaction`
   - [ ] 设置 `triggerThreshold`（推荐 8K Token）
   - [ ] 设置 `targetTokens`（推荐 5K）

4. **会话清理** ✓
   - [ ] 配置 `expiration` 策略
   - [ ] 定期执行 `openclaw cleanup`

5. **成本分析** ✓
   - [ ] 每周运行 `/cost` 命令，审视成本趋势
   - [ ] 识别最耗资源的 Agent 与操作
   - [ ] 制定优化计划

6. **模型分层** ✓
   - [ ] 为不同复杂度的任务选择合适的模型
   - [ ] 或启用自动复杂度检测

完成以上配置，可实现**可观、可控、可优化**的成本管理体系。
