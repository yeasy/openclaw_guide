## 14.1 Token 与上下文成本

Token 消耗是 Agent 系统最直接的成本驱动因素。本节从 Token 的流入与流出、精简策略、上下文压缩等维度，提供基于 OpenClaw 配置的系统化优化方案。

### 14.1.1 Token 流动的三个阶段

一个完整的 Agent 对话中，Token 消耗分为三个阶段：
```yaml
阶段 1: 输入 Token
  系统提示 + 上下文窗口 + 用户消息
  ↓
[LLM 推理]
  ↓
阶段 2: 输出 Token
  模型生成的回答 + 思考过程
  ↓
阶段 3: 下一轮消息历史
  将本轮对话写入会话，成为下一轮的上下文
  （→ 回到阶段 1）
```

**关键观察**：会话越长，上下文窗口越大，每一轮的输入 Token 越多。这形成**成本加速**：第 10 轮对话的输入 Token 数往往是第 1 轮的 5-10 倍。

### 14.1.2 输入 Token 优化

#### 1. 系统提示精简

系统提示（System Prompt）是每轮推理都必须发送的固定成本。

**优化前**（常见冗余写法）：
```text
你是一个客服助手。你的职责是帮助用户解决问题。
你应该始终保持友好和专业的态度。
你不应该提供医疗建议。
你应该尽快解决用户的问题。
... [还有 20 行类似描述] ...
```

<!-- ~2000 字 → 约 500 Token -->
**优化后**（精简版）：
```text
客服助手。友好、专业、拒绝医疗建议。
```

<!-- ~25 字 → 约 10 Token -->
**收益**：节省 490 Token/轮 × 1000 轮/月 = 49 万 Token。若按本章后文示例的 Claude Sonnet 4.6 输入价 `$3 / 1M` 粗算，约为 `$1.47 / 月`。

#### 2. 按 Agent 差异化工具定义

不是所有 Agent 都需要所有工具。精简工具列表可减少提示词中的工具定义部分。

**优化前**（通用工具列表）：
```jsonc
{
  "tools": [
    "search_web", "send_email", "create_calendar_event",
    "read_file", "write_file", "delete_file",
    "query_database", "execute_sql", "call_api"
    // ... 以下省略 20+ 个工具
  ]
}
```

**优化后**（按 Agent 分配）：
```json
{
  "agents": {
    "support_agent": {
      "tools": ["search_knowledge_base", "send_email", "create_ticket"]
    },
    "data_analyst": {
      "tools": ["query_database", "execute_sql", "plot_chart"]
    }
  }
}
```

#### 3. 上下文窗口预算与动态截断

即使有 100K Token 的窗口，也不应将所有历史都塞入每个请求。关键是**选择性加载**。

按当前 OpenClaw schema，旧资料里常见的根级 `context.maxTokens`、`pruningStrategy`、`recency_weight` 等字段不应再当作官方配置项。OpenClaw 没有公开 `maxInputTokens` 这类总输入 Token 硬上限；更可执行的做法是用 `contextPruning` 裁剪旧工具结果缓存，并通过模型选择、提示词长度和工具输出约束控制请求规模。

示例：
```json
{
  "agents": {
    "defaults": {
      "contextPruning": {
        "mode": "cache-ttl",
        "ttl": "1h",
        "softTrim": { "maxChars": 4000 },
        "hardClear": { "enabled": true }
      }
    }
  }
}
```

至于“能从 15K 降到 8K”这类数字，只有在你自己的真实工作负载上做过测量后才值得写进运维基线。

### 14.1.3 输出 Token 优化

#### 1. 限制输出长度

对于不需要长回答的任务，明确限制输出长度。

配置示例：
```json
{
  "agents": {
    "qa_agent": {
      "maxCompletionTokens": 500
    },
    "research_agent": {
      "maxCompletionTokens": 2000
    }
  }
}
```

#### 2. 结构化输出格式

使用 JSON Schema 或其他结构化格式，可让模型生成更精简的回答。

**优化前**（自由文本）：
```text
用户询问："我的包裹到哪了？"
模型回答："您好！您的包裹当前位于中转站。根据我们的追踪系统，
您的订单（订单号：12345678）已于今日上午 10:30 到达北京中转站。
预计明日下午送达。如有任何问题，请随时联系我们。"
```
（约 150 Token）

**优化后**（结构化）：
```json
{
  "status": "in_transit",
  "location": "Beijing Hub",
  "eta": "2026-03-23T18:00Z",
  "order_id": "12345678"
}
```
（约 30 Token）

### 14.1.4 上下文压缩与裁剪策略

当会话变得很长时，必须主动压缩以避免成本爆炸。

#### 策略 1：消息合并（Compaction）

将多个老旧消息合并为单个摘要消息。

**压缩后**（1 条摘要消息，200 Token）：
```text
[之前讨论摘要] 用户询问了产品 A 的特性、价格（$99）、优惠（20% 折扣）、
交付时间（3-5 天）。用户表示感兴趣。上一条消息：用户问何时有货。
```

如果你需要显式指定 compaction 模型，应把配置挂在 `agents.defaults.compaction` 下，例如：
```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "model": "anthropic/claude-haiku-4-5"
      }
    }
  }
}
```

#### 策略 2：优先级裁剪（Pruning）

当前更准确的说法是：pruning 主要用于**在模型调用前裁剪旧工具结果缓存**，而不是删除磁盘上的完整会话历史。

它适合做的事情包括：
- 保留最近仍有引用价值的工具结果
- 清掉已过 TTL 的旧工具结果缓存
- 限制同名工具结果在上下文中的回注数量

而“把整段会话总结成摘要”应由 compaction 负责，不应混写成 pruning 规则。

### 14.1.5 模型选择的成本影响

不同的模型在 Token 计价与效率上差异巨大。

**成本对比**（2026 年 4 月定价）：

| 模型 | 输入单价 | 输出单价 | 特性 | 适用场景 |
|------|---------|---------|------|---------|
| Claude Haiku 4.5 | $1/1M | $5/1M | 快速、便宜 | 简单分类、摘要 |
| Claude Sonnet 4.6 | $3/1M | $15/1M | 平衡 | 通用助手 |
| Claude Opus 4.6 | $5/1M | $25/1M | 强推理能力 | 复杂多步任务 |
| Claude Opus 4.7 | $5/1M | $25/1M | 最强推理、1M 上下文元数据 | 高难度软件工程、复杂多步任务 |
| GPT-5.5 | $5/1M | $30/1M | OpenAI 最新旗舰 | 高难度推理、复杂任务 |
| GPT-5.4 | $2.50/1M | $15/1M | OpenAI 前沿主线 | 通用、复杂推理 |
| GPT-5.4 mini | $0.75/1M | $4.50/1M | 高性价比 | 高并发、低延迟 |
| GPT-5.4 nano | $0.20/1M | $1.25/1M | 最低成本 | 简单任务、高频调用 |

典型分层：简单问题（80%）→ Haiku，中等（15%）→ Sonnet，复杂（5%）→ Opus，可降低平均成本 50-70%。

> **注意**：不要把固定 tokenizer 膨胀倍率写入 OpenClaw 运维基线；当前 OpenClaw 代码没有内置 Opus 4.7 tokenizer multiplier。请用 `usage-cost` 导出、provider billing 与真实任务样本测量有效成本，再决定是否通过 Prompt Caching、Batch Processing 或模型分层抵消成本。


```bash

```

### 14.1.7 成本优化检查清单

在优化 Token 成本时，按以下顺序检查：

1. **系统提示长度** < 100 字？ □
   - 若否：精简到 50 字以内
2. **工具列表** < 5 个工具/Agent？ □
   - 若否：为不同 Agent 分配差异化工具
3. **大型工具结果是否被裁剪**？ □
   - 若否：启用 `contextPruning` 并限制工具输出字符数
4. **消息历史** < 20 条？ □
   - 若否：启用 `compaction`，压缩到 10 条 + 摘要
5. **输出限制** 已配置？ □
   - 若否：为各 Agent 设置 `maxCompletionTokens`
6. **模型选择** 有分层吗？ □
   - 若否：为简单任务用 Haiku，复杂用 Sonnet

完成以上 6 项优化，通常可以将 Token 成本降低 40-60%。
