## 14.1 Token 与上下文成本

Token 消耗是 Agent 系统最直接的成本驱动因素。本节从 Token 的流入与流出、精简策略、上下文压缩等维度，提供基于 OpenClaw 配置的系统化优化方案。

### 14.1.1 Token 流动的三个阶段

一个完整的 Agent 对话中，Token 消耗分为三个阶段：
```yaml
阶段 1: 输入 Token
  系统提示 + 上下文窗口 + 用户消息
  ↓
[LLM 推理]
  ↓
阶段 2: 输出 Token
  模型生成的回答 + 思考过程
  ↓
阶段 3: 下一轮消息历史
  将本轮对话写入会话，成为下一轮的上下文
  （→ 回到阶段 1）
```

**关键观察**：会话越长，上下文窗口越大，每一轮的输入 Token 越多。这形成**成本加速**：第 10 轮对话的输入 Token 数往往是第 1 轮的 5-10 倍。

### 14.1.2 输入 Token 优化

#### 1. 系统提示精简

系统提示（System Prompt）是每轮推理都必须发送的固定成本。

**优化前**（常见冗余写法）：
```text
你是一个客服助手。你的职责是帮助用户解决问题。
你应该始终保持友好和专业的态度。
你不应该提供医疗建议。
你应该尽快解决用户的问题。
... [还有 20 行类似描述] ...
```

<!-- ~2000 字 → 约 500 Token -->
**优化后**（精简版）：
```text
客服助手。友好、专业、拒绝医疗建议。
```

<!-- ~25 字 → 约 10 Token -->
**收益**：节省 490 Token/轮 × 1000 轮/月 = 49 万 Token。若按本章后文示例的 Claude Sonnet 4.6 输入价 `$3 / 1M` 粗算，约为 `$1.47 / 月`。

#### 2. 按 Agent 差异化工具定义

不是所有 Agent 都需要所有工具。精简工具列表可减少提示词中的工具定义部分。

**优化前**（通用工具列表）：
```jsonc
{
  "tools": [
    "search_web", "send_email", "create_calendar_event",
    "read_file", "write_file", "delete_file",
    "query_database", "execute_sql", "call_api"
    // ... 以下省略 20+ 个工具
  ]
}
```

**优化后**（按 Agent 分配）：
```json
{
  "agents": {
    "support_agent": {
      "tools": ["search_knowledge_base", "send_email", "create_ticket"]
    },
    "data_analyst": {
      "tools": ["query_database", "execute_sql", "plot_chart"]
    }
  }
}
```

#### 3. 上下文窗口预算与动态截断

即使有 100K Token 的窗口，也不应将所有历史都塞入每个请求。关键是**选择性加载**。

配置示例：
```json
{
  "context": {
    "maxTokens": 10000,
    "pruningStrategy": "hybrid",
    "recency_weight": 0.6,
    "relevance_weight": 0.4,
    "minHistoryMessages": 3,
    "maxHistoryMessages": 20
  }
}
```

通过智能去重，通常可将上下文从 15K 降至 8K Token，同时保持回答质量。

### 14.1.3 输出 Token 优化

#### 1. 限制输出长度

对于不需要长回答的任务，明确限制输出长度。

配置示例：
```json
{
  "agents": {
    "qa_agent": {
      "maxCompletionTokens": 500
    },
    "research_agent": {
      "maxCompletionTokens": 2000
    }
  }
}
```

#### 2. 结构化输出格式

使用 JSON Schema 或其他结构化格式，可让模型生成更精简的回答。

**优化前**（自由文本）：
```text
用户询问："我的包裹到哪了？"
模型回答："您好！您的包裹当前位于中转站。根据我们的追踪系统，
您的订单（订单号：12345678）已于今日上午 10:30 到达北京中转站。
预计明日下午送达。如有任何问题，请随时联系我们。"
```
（约 150 Token）

**优化后**（结构化）：
```json
{
  "status": "in_transit",
  "location": "Beijing Hub",
  "eta": "2026-03-23T18:00Z",
  "order_id": "12345678"
}
```
（约 30 Token）

### 14.1.4 上下文压缩与裁剪策略

当会话变得很长时，必须主动压缩以避免成本爆炸。

#### 策略 1：消息合并（Compaction）

将多个老旧消息合并为单个摘要消息。

**压缩后**（1 条摘要消息，200 Token）：
```text
[之前讨论摘要] 用户询问了产品 A 的特性、价格（$99）、优惠（20% 折扣）、
交付时间（3-5 天）。用户表示感兴趣。上一条消息：用户问何时有货。
```

**配置示例**：
```json
{
  "context": {
    "compaction": {
      "enabled": true,
      "triggerThreshold": 8000,
      "preserveRecentMessages": 5,
      "summarizationModel": "claude-haiku-4-5"
    }
  }
}
```

#### 策略 2：优先级裁剪（Pruning）

删除低优先级的消息（如无关、陈旧的对话）。

**规则示例**：
- 保留最近 5 条消息（总是需要最新上下文）
- 保留包含“重要”标签的任何消息
- 删除超过 7 天前的调试日志消息
- 删除被标记为“解决”的问题讨论

### 14.1.5 模型选择的成本影响

不同的模型在 Token 计价与效率上差异巨大。

**成本对比**（2026 年 4 月定价）：

| 模型 | 输入单价 | 输出单价 | 特性 | 适用场景 |
|------|---------|---------|------|---------|
| Claude Haiku 4.5 | $1/1M | $5/1M | 快速、便宜 | 简单分类、摘要 |
| Claude Sonnet 4.6 | $3/1M | $15/1M | 平衡 | 通用助手 |
| Claude Opus 4.6 | $5/1M | $25/1M | 强推理能力 | 复杂多步任务 |
| Claude Opus 4.7 | $5/1M | $25/1M | 最强推理，新 tokenizer（+35% tokens） | 高难度软件工程、复杂多步任务 |
| GPT-5.5 | $5/1M | $30/1M | OpenAI 最新旗舰 | 高难度推理、复杂任务 |
| GPT-5.4 | $2.50/1M | $15/1M | OpenAI 前沿主线 | 通用、复杂推理 |
| GPT-5.4 mini | $0.75/1M | $4.50/1M | 高性价比 | 高并发、低延迟 |
| GPT-5.4 nano | $0.20/1M | $1.25/1M | 最低成本 | 简单任务、高频调用 |

典型分层：简单问题（80%）→ Haiku，中等（15%）→ Sonnet，复杂（5%）→ Opus，可降低平均成本 50-70%。

> **注意：Claude Opus 4.7 的 Tokenizer 变化**：Opus 4.7 采用了新的 tokenizer，token 膨胀率因内容类型而异：英文纯文本接近 1.0x，代码和结构化数据可达 1.35x。虽然 per-token 定价保持不变（$5/$25），但复杂技术内容的实际有效成本可能显著上升。可通过启用 Prompt Caching（节省高达 90%）或 Batch Processing（节省 50%）来部分抵消这一成本增加。

### 14.1.6 Token 成本实时监控

使用 OpenClaw 内置命令监控 Token 消耗：

```bash
# 查看当前会话的 Token 统计
openclaw status --session=<session_id>
```

```bash
# 导出成本趋势（JSON 格式，便于制图）
openclaw cost --period=7d --format=json > cost_trend.json
```

### 14.1.7 成本优化检查清单

在优化 Token 成本时，按以下顺序检查：

1. **系统提示长度** < 100 字？ □
   - 若否：精简到 50 字以内
2. **工具列表** < 5 个工具/Agent？ □
   - 若否：为不同 Agent 分配差异化工具
3. **上下文预算** < 10K Token？ □
   - 若否：启用 `contextPruning`，降到 8K
4. **消息历史** < 20 条？ □
   - 若否：启用 `compaction`，压缩到 10 条 + 摘要
5. **输出限制** 已配置？ □
   - 若否：为各 Agent 设置 `maxCompletionTokens`
6. **模型选择** 有分层吗？ □
   - 若否：为简单任务用 Haiku，复杂用 Sonnet

完成以上 6 项优化，通常可以将 Token 成本降低 40-60%。
