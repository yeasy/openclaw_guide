## 14.1 Token 与上下文成本

Token 消耗是 Agent 系统最直接的成本驱动因素。本节从 Token 的流入与流出、精简策略、上下文压缩等维度，提供基于 OpenClaw 配置的系统化优化方案。

### 14.1.1 Token 流动的三个阶段

一个完整的 Agent 对话中，Token 消耗分为三个阶段：

```
阶段 1: 输入 Token
  系统提示 + 上下文窗口 + 用户消息
  ↓
[LLM 推理]
  ↓
阶段 2: 输出 Token
  模型生成的回答 + 思考过程
  ↓
阶段 3: 下一轮消息历史
  将本轮对话写入会话，成为下一轮的上下文
  （→ 回到阶段 1）
```

**关键观察**：会话越长，上下文窗口越大，每一轮的输入 Token 越多。这形成**成本加速**：第 10 轮对话的输入 Token 数往往是第 1 轮的 5-10 倍。

### 14.1.2 输入 Token 优化

#### 1. 系统提示精简

系统提示（System Prompt）是每轮推理都必须发送的固定成本。

**优化前**（常见冗余写法）：
```
你是一个客服助手。你的职责是帮助用户解决问题。
你应该始终保持友好和专业的态度。
你不应该提供医疗建议。
你应该尽快解决用户的问题。
... [还有 20 行类似描述] ...
```

**字符数**：~2000 字 → 约 500 Token

**优化后**（精简版）：
```
客服助手。友好、专业、拒绝医疗建议。
```

**字符数**：~25 字 → 约 10 Token

**收益**：节省 490 Token/轮 × 1000 轮/月 = 49 万 Token ≈ $14.7/月

#### 2. 按 Agent 差异化工具定义

不是所有 Agent 都需要所有工具。精简工具列表可减少提示词中的工具定义部分。

**优化前**（通用工具列表）：
```json
{
  "tools": [
    "search_web", "send_email", "create_calendar_event",
    "read_file", "write_file", "delete_file",
    "query_database", "execute_sql", "call_api",
    ... [20+ 个工具]
  ]
}
```

**优化后**（按 Agent 分配）：
```json
{
  "agents": {
    "support_agent": {
      "tools": ["search_knowledge_base", "send_email", "create_ticket"]
    },
    "data_analyst": {
      "tools": ["query_database", "execute_sql", "plot_chart"]
    }
  }
}
```

**收益**：每个 Agent 的工具定义从 20 个减到 3 个 → 节省 400-600 Token/轮

#### 3. 上下文窗口预算与动态截断

即使有 100K Token 的窗口，也不应将所有历史都塞入每个请求。关键是**选择性加载**。

配置示例：
```json
{
  "context": {
    "maxTokens": 10000,
    "pruningStrategy": "hybrid",
    "recency_weight": 0.6,
    "relevance_weight": 0.4,
    "minHistoryMessages": 3,
    "maxHistoryMessages": 20
  }
}
```

**策略解读**：
- `maxTokens: 10000`：单个会话的最大上下文 Token
- `pruningStrategy: "hybrid"`：结合时间与语义相关性去掉旧消息
- `recency_weight: 0.6`：优先保留最近消息（60% 权重）
- `relevance_weight: 0.4`：也保留语义相关的旧消息（40% 权重）
- `minHistoryMessages: 3`：至少保留 3 条消息（保证连贯性）
- `maxHistoryMessages: 20`：最多保留 20 条消息（避免过度膨胀）

**实际效果**：通过智能去重，通常可将上下文从 15K Token 降到 8K Token，同时保持回答质量 >95%。

### 14.1.3 输出 Token 优化

#### 1. 限制输出长度

对于不需要长回答的任务，明确限制输出长度。

配置示例：
```json
{
  "agents": {
    "qa_agent": {
      "maxCompletionTokens": 500
    },
    "research_agent": {
      "maxCompletionTokens": 2000
    }
  }
}
```

**权衡**：
- 限制过严（如 100 Token）：可能导致回答被截断，质量下降
- 合理限制（如 500-800 Token）：足够表达完整答案，同时控制成本
- 无限制（默认行为）：可能产生 5000+ Token 的冗长回答

#### 2. 结构化输出格式

使用 JSON Schema 或其他结构化格式，可让模型生成更精简的回答。

**优化前**（自由文本）：
```
用户询问："我的包裹到哪了？"
模型回答："您好！您的包裹当前位于中转站。根据我们的追踪系统，
您的订单（订单号：12345678）已于今日上午 10:30 到达北京中转站。
预计明日下午送达。如有任何问题，请随时联系我们。"
（约 150 Token）
```

**优化后**（结构化）：
```json
{
  "status": "in_transit",
  "location": "Beijing Hub",
  "eta": "2026-03-23T18:00Z",
  "order_id": "12345678"
}
（约 30 Token）
```

**收益**：节省 120 Token/回答 × 100 回答/天 = 12K Token/天 ≈ $1.8/天

### 14.1.4 上下文压缩与裁剪策略

当会话变得很长时，必须主动压缩以避免成本爆炸。

#### 策略 1：消息合并（Compaction）

将多个老旧消息合并为单个摘要消息。

**原始消息序列**（10 条消息，总 5K Token）：
```
User: "告诉我关于产品 A 的一切"
Assistant: "产品 A 是...（详细说明）"
User: "它的价格是多少？"
Assistant: "价格是 $99..."
User: "有优惠吗？"
Assistant: "是的，现在有 20% 折扣..."
... [7 条更多消息，讨论功能、交付等]
```

**压缩后**（1 条摘要消息，200 Token）：
```
[之前讨论摘要] 用户询问了产品 A 的特性、价格（$99）、优惠（20% 折扣）、
交付时间（3-5 天）。用户表示感兴趣。上一条消息：用户问何时有货。
```

**配置示例**：
```json
{
  "context": {
    "compaction": {
      "enabled": true,
      "triggerThreshold": 8000,
      "preserveRecentMessages": 5,
      "summarizationModel": "gpt-3.5-turbo"
    }
  }
}
```

#### 策略 2：优先级裁剪（Pruning）

删除低优先级的消息（如无关、陈旧的对话）。

**规则示例**：
- 保留最近 5 条消息（总是需要最新上下文）
- 保留包含"重要"标签的任何消息
- 删除超过 7 天前的调试日志消息
- 删除被标记为"解决"的问题讨论

### 14.1.5 模型选择的成本影响

不同的模型在 Token 计价与效率上差异巨大。

**成本对比**（假设 2026 年 3 月定价）：

| 模型 | 输入单价 | 输出单价 | 特性 | 适用场景 |
|------|---------|---------|------|---------|
| Claude 3.5 Haiku | $0.80/100K | $4/100K | 快速、便宜 | 简单分类、摘要 |
| Claude 3.5 Sonnet | $3/100K | $15/100K | 平衡 | 通用助手 |
| Claude 3 Opus | $15/100K | $75/100K | 强大、贵 | 复杂推理 |
| GPT-4 Turbo | $10/100K | $30/100K | OpenAI 方案 | 需要 GPT 特定功能 |

**选择策略**：
```
简单问题（80%）→ Haiku（成本最低）
中等复杂度（15%）→ Sonnet（平衡）
复杂推理（5%）→ Opus（质量优先）
```

通过分层路由，可将平均成本降低 50-70%。

### 14.1.6 Token 成本实时监控

使用 OpenClaw 内置命令监控 Token 消耗：

```bash
# 查看当前会话的 Token 统计
openclaw status --session=<session_id>

# 输出例：
# Session: user_123
# Messages: 42
# Input Tokens: 18500 ($0.555)
# Output Tokens: 4200 ($0.630)
# Total Cost: $1.185
# Context Compression: 65% reduction applied
```

```bash
# 导出成本趋势（JSON 格式，便于制图）
openclaw cost --period=7d --format=json > cost_trend.json
```

### 14.1.7 成本优化检查清单

在优化 Token 成本时，按以下顺序检查：

1. **系统提示长度** < 100 字？ □
   - 若否：精简到 50 字以内
2. **工具列表** < 5 个工具/Agent？ □
   - 若否：为不同 Agent 分配差异化工具
3. **上下文预算** < 10K Token？ □
   - 若否：启用 `contextPruning`，降到 8K
4. **消息历史** < 20 条？ □
   - 若否：启用 `compaction`，压缩到 10 条 + 摘要
5. **输出限制** 已配置？ □
   - 若否：为各 Agent 设置 `maxCompletionTokens`
6. **模型选择** 有分层吗？ □
   - 若否：为简单任务用 Haiku，复杂用 Sonnet

完成以上 6 项优化，通常可以将 Token 成本降低 40-60%。
