## 13.1 实战案例：企业飞书群工作助手

### 13.1.1 问题定义与场景约束

一个 30 人规模的工程团队日常通过飞书（Lark）协作，希望在群里有一个智能助手，具备以下能力：

- 回答关于内部文档和代码仓库的问题
- 总结每日 GitHub PR 活动
- 通过自然语言创建和查询 Jira 工单
- 根据用户角色实施权限控制（工程师 vs 访客）

约束条件：必须在现有飞书群内运行，无需新增基础设施（仅通过 OpenClaw Gateway）。

### 13.1.2 最小闭环：单工具单渠道跑通

从最简单的配置开始，验证群机器人的基本工作流程。我们用一个知识库搜索工具、一个飞书渠道、一份可以直接粘贴运行的配置文件。

下面是完整的 `openclaw.json` 最小化配置：

```json
{
  "profile": "default",
  "version": "1.0.0",
  "agents": {
    "group_assistant": {
      "name": "群工作助手",
      "instructions": "你是企业工程团队的群助手。\n用户会在飞书群里提问，你的职责是：\n1. 搜索内部知识库，回答常见问题\n2. 保持回复简洁、专业\n3. 如果知识库里没有答案，诚实地说 \"我找不到相关文档\"",
      "model": {
        "type": "openai",
        "name": "gpt-4o-mini",
        "parameters": {
          "temperature": 0.7,
          "max_tokens": 800
        }
      },
      "tools": [
        "knowledge_base_search"
      ],
      "memory": {
        "type": "context_window",
        "max_messages": 10
      }
    }
  },
  "tools": {
    "knowledge_base_search": {
      "type": "builtin:knowledge_base",
      "description": "在内部知识库中搜索文档和代码注释",
      "config": {
        "index_path": "./kb/docs",
        "similarity_threshold": 0.6
      }
    }
  },
  "channels": {
    "lark": {
      "type": "feishu",
      "enabled": true,
      "config": {
        "app_id": "${LARK_APP_ID}",
        "app_secret": "${LARK_APP_SECRET}",
        "domain": "feishu.cn",
        "group_mode": true,
        "mention_required": false
      }
    }
  },
  "pairing": {
    "strategy": "first_time_approve",
    "ttl_minutes": 10
  }
}
```

按照[第七章 7.2 飞书专项接入指南](../07_multi_agent/7.2_lark_integration.md)的步骤完成飞书应用配置和 OpenClaw 端的绑定后，运行网关：

```bash
openclaw gateway start
```

然后在飞书群里发送一条消息，机器人应该会搜索知识库并回复相关内容。

验证步骤：发送一条消息到群里（例如“如何重置密码”），确认机器人返回了知识库中的相关答案。如果没有回复，检查[第七章 7.2](../07_multi_agent/7.2_lark_integration.md)中的“常见失败点速查”。

### 13.1.3 关键配置拆解

现在逐步添加复杂功能。每一层都只展示增量部分，完整配置见前一小节。

#### 1. 增加工具：GitHub PR 分析与 Jira 工单管理

修改上面配置中的 `agents.group_assistant.tools` 列表，添加两个工具：

```json
"tools": [
  "knowledge_base_search",
  "github_pr_analyzer",
  "jira_issue_manager"
]
```

然后在 `tools` 对象中补充定义：

```json
"tools": {
  "knowledge_base_search": { },
  "github_pr_analyzer": {
    "type": "builtin:github",
    "description": "分析 GitHub 仓库中过去 24 小时的 PR 活动",
    "config": {
      "token": "${GITHUB_TOKEN}",
      "owner": "my-company",
      "repos": ["backend", "frontend", "infra"],
      "lookback_hours": 24
    }
  },
  "jira_issue_manager": {
    "type": "builtin:jira",
    "description": "创建和查询 Jira 工单",
    "config": {
      "instance": "my-company.atlassian.net",
      "token": "${JIRA_TOKEN}",
      "project_key": "ENG"
    }
  }
}
```

详见[第五章 工具入门](../05_tools_skills/README.md)。

#### 2. 添加权限控制：基于角色的访问策略

在配置顶层添加 `profiles` 对象，定义角色和权限清单：

```json
"profiles": {
  "engineer": {
    "clearance": 3,
    "allowed_tools": ["knowledge_base_search", "github_pr_analyzer", "jira_issue_manager"],
    "description": "工程师，可以使用所有工具"
  },
  "guest": {
    "clearance": 1,
    "allowed_tools": ["knowledge_base_search"],
    "description": "访客，只能搜索知识库"
  }
}
```

然后在 `agents.group_assistant` 中添加守卫规则：

```json
"guardrails": {
  "enforce_profile": true,
  "default_profile": "guest",
  "tool_deny": [
    {
      "tool": "github_pr_analyzer",
      "condition": "profile != 'engineer'",
      "reason": "仅工程师可以查看 PR 活动"
    },
    {
      "tool": "jira_issue_manager",
      "condition": "profile != 'engineer'",
      "reason": "仅工程师可以操作工单"
    }
  ]
}
```

详见[第十一章 11.4 守卫与隔离](../11_reliability_security/11.4_guardrails.md)。

#### 3. 添加定时任务：每日 PR 摘要

在配置顶层添加 `cron_tasks` 对象：

```json
"cron_tasks": {
  "daily_pr_summary": {
    "schedule": "0 16 * * *",
    "agent": "group_assistant",
    "tool": "github_pr_analyzer",
    "channel": "lark",
    "instruction": "生成过去 24 小时的 PR 活动摘要，列出：1) 新开的 PR 数量、2) 已合并的 PR 数量、3) 待审核 PR 列表。用简洁的 Markdown 格式。",
    "timezone": "Asia/Shanghai"
  }
}
```

这会在每天下午 16:00（北京时间）自动向飞书群发送 PR 摘要。详见[第八章 自动化与定时任务](../08_automation_ops/README.md)。

### 13.1.4 验收清单与常见失败点

逐项验证配置的正确性：

1. **机器人在群里有回复** → 在飞书群中 @ 机器人或直接发消息，等待 3 秒内收到回复
2. **知识库搜索有结果** → 提问一个文档中确实存在的问题（例如“API 文档在哪里”），确认搜索返回相关内容
3. **权限控制生效** → 用访客账户在群里发送“请分析最近的 PR”，验证机器人回复“您没有权限使用此功能”或类似拒绝提示
4. **定时任务正常触发** → 检查日志确认任务在预期时间（北京时间 16:00）执行

常见失败及排查方案：

| 问题 | 可能原因 | 排查步骤 |
|------|--------|--------|
| “机器人不回复” | 飞书 Webhook URL 配置错误或验证令牌不匹配 | 查看[第七章 7.2](../07_multi_agent/7.2_lark_integration.md)“防坑预警”；检查 `openclaw.json` 中 `LARK_APP_ID` 和 `LARK_APP_SECRET` 环境变量是否正确设置 |
| “知识库没有结果” | 索引路径不存在或文档格式不兼容 | 验证 `tools.knowledge_base_search.config.index_path` 目录存在；检查文档是否为支持的格式（txt、md、pdf） |
| “访客仍能调用高权限工具” | 守卫规则优先级或条件语法错误 | 运行 `openclaw status --deep` 查看完整的守卫配置；确认 `tools.deny` 规则中 `condition` 的语法正确 |
| “定时任务不执行” | 时区配置错误或 cron 表达式语法错误 | 检查 `timezone` 字段是否为有效的 IANA 时区名称（如 `Asia/Shanghai`）；验证 `schedule` 是标准 5 字段 cron 格式 |

### 13.1.5 扩展到生产时再加什么

当从测试环境迁移到生产时，需要补充以下几个方面（配置细节参见相应章节）：

**Kubernetes 部署** - 不再用 `openclaw gateway start` 本地运行，而是部署成 Pod，支持多副本和自动扩展。详见[第十二章 12.4 生产落地蓝图](../12_extension_engineering/12.4_production_blueprint.md)。

**监控和告警** - 接入 Prometheus 或 DataDog 监控机器人的响应时间、错误率、工具调用频率。详见[第十二章 12.4.5 监控与告警](../12_extension_engineering/12.4_production_blueprint.md)。

**多渠道支持** - 同时接入 Slack、钉钉等平台。参考[第七章](../07_multi_agent/README.md)中各渠道的集成指南。

**审计日志** - 记录所有工具调用、用户操作和权限检查结果。详见[第十一章 11.4 守卫与隔离](../11_reliability_security/11.4_guardrails.md)。

完整的 Kubernetes 部署配置与监控告警设置，参见[第十二章生产落地蓝图](../12_extension_engineering/12.4_production_blueprint.md)。
