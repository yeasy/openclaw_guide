## 9.4 事件幂等与一致性保障

在分布式系统中，网络是不可靠的：消息可能丢失、延迟、重复。OpenClaw 的数据平面与控制平面必须设计成"**至少一次（at-least-once）**"的语义，并通过幂等机制确保重复执行不会产生副作用。本节深入事件驱动系统中的一致性问题与解决方案。

### 9.4.1 为什么"至少一次"是常态

网络请求有三种可能的结果：

| 结果 | 含义 | 设备侧感受 |
|-----|------|----------|
| **Success** | 服务器收到且处理成功 | 等待 SUCC 响应 |
| **Failure** | 服务器收到但处理失败 | 等待 ERR 响应 |
| **Unknown** | 请求可能丢失，或成功了但响应丢失 | 超时，不知道发生了什么 |

当遇到"Unknown"时，一个**可靠**的系统会选择**重试**：

```
发送 event_A (version=1)
  ↓
超时，无响应
  ↓
重试 event_A (version=1)
  ↓
收到 SUCC
```

问题来了：如果第一次请求其实成功了，第二次重试会导致 `event_A` 被处理两次。从服务器的视角，它看到了同一个事件，需要识别出这是重复，而不是两个不同的操作。

**这正是幂等性的定义**：无论执行多少次，结果都是一样的。

### 9.4.2 幂等性的三个层次

#### 1. 天然幂等的操作

某些操作天生就是幂等的，无需特殊处理：

```
PUT /user/profile { "name": "Alice" }  # 无论调用多少次，结果都是 name=Alice
GET /user/profile                      # 无论调用多少次，都返回相同结果
DELETE /file/temp.log                  # 第一次删除成功，第二次报错"文件不存在"
                                        # 但从业务角度，文件已删除，符合幂等性
```

#### 2. 条件幂等的操作

某些操作需要条件检查才能保证幂等性：

```
POST /account/transfer {
  "from": 100,
  "to": 200,
  "amount": 50,
  "idempotency_key": "txn_abc123"
}
```

如果第一次转账成功，响应中会包含 `idempotency_key` 的结果。设备重试时，服务器查询"idempotency_key = txn_abc123"的历史，发现已处理，直接返回上次的结果，而不是再转一次钱。

#### 3. 非幂等的操作

某些操作本质上无法幂等（不应该在分布式系统中直接使用）：

```
POST /counter/increment { "value": 1 }  # 每次都会 +1，无法幂等
```

对于非幂等操作，系统需要提供"分布式锁"或"单步提交"机制来确保只执行一次。在 OpenClaw 中，这通常由控制平面通过 **事务边界** 来处理。

### 9.4.3 幂等密钥（Idempotency Key）

OpenClaw 采用**幂等密钥**机制来支持条件幂等：

#### 生成与传递

设备在发送任何会改变状态的请求时，必须附带一个唯一的 `idempotency_key`：

```json
{
  "type": "TOOL_RESULT",
  "session_id": "sess_7f9e2a",
  "tool_name": "send_email",
  "result": { "email_id": "msg_456" },
  "idempotency_key": "ack_20260322_1712_001"
}
```

这个密钥的构成通常是：

```
idempotency_key = [prefix]_[timestamp]_[sequence]_[hash]
```

例如：

- `prefix` = "ack"（确认类）、"tool"（工具结果类）等
- `timestamp` = 消息发送时的时间戳
- `sequence` = 在该连接上的消息序号（递增）
- `hash` = 内容哈希（可选，用于检测重复内容变化）

#### 服务器端处理

Gateway 的控制平面维护一个"幂等密钥表"：

```
idempotency_table[device_id][idempotency_key] = {
  "status": "success",
  "result": { ... },
  "timestamp": 1711100000,
  "expires_at": 1711200000
}
```

处理流程：

```
received_request = parse(message)
key = received_request.idempotency_key

if key in idempotency_table:
    # 重复请求
    cached = idempotency_table[key]
    if cached.status == "success":
        return cached.result  # 直接返回缓存结果，不再执行
    elif cached.status == "processing":
        return error("still_processing")  # 仍在处理，让设备等待
    elif cached.status == "failure":
        return error("previous_failure")  # 上次失败，返回同样的错误
else:
    # 新请求，执行并记录
    idempotency_table[key] = { "status": "processing" }
    try:
        result = execute(received_request)
        idempotency_table[key] = { "status": "success", "result": result }
        return result
    except Exception as e:
        idempotency_table[key] = { "status": "failure", "error": e.msg }
        return error(e.msg)
```

#### 密钥过期与清理

幂等密钥需要被保留足够长的时间，以应对迟到的重试，但不能无限保留（浪费存储）：

- **保留期**：通常为 24 小时或配置参数 `idempotency_key_ttl`
- **清理策略**：后台定时任务删除超过 TTL 的条目，或使用 Redis 的自动过期

### 9.4.4 事件顺序保障

在消息可能乱序的网络中，必须明确定义"事件顺序"的含义。

#### 顺序保障的三个级别

| 级别 | 定义 | 成本 | 适用场景 |
|-----|------|------|--------|
| **无序** | 事件可以任意顺序执行 | 最低 | 多个独立工具调用 |
| **会话级序列化** | 同一会话内的事件必须按顺序执行 | 中等 | 大多数任务（对话、工具链） |
| **全局顺序** | 所有用户的所有事件按全局时间戳排序 | 最高 | 财务交易、审计日志 |

OpenClaw 默认采用**会话级序列化**，原因是：

- 不同会话是独立的，无需全局排序
- 同一会话内，事件必须依序执行（如"获取邮件"→"总结"→"发送"）
- 这样既保证了正确性，又避免了全局排序的性能开销

#### 实现机制：序列号与乱序重组

控制平面为每个会话维护一个"期望序列号"：

```
session.expected_seq = 1

received_event:
  if event.seq == expected_seq:
      # 顺序事件，立即执行
      execute(event)
      expected_seq += 1
      # 处理缓冲区中乱序的事件
      while (expected_seq in out_of_order_buffer):
          execute(out_of_order_buffer[expected_seq])
          delete out_of_order_buffer[expected_seq]
          expected_seq += 1
  elif event.seq > expected_seq:
      # 乱序事件，缓冲等待
      out_of_order_buffer[event.seq] = event
  else:
      # 重复事件，已处理过，忽略
      skip(event)
```

**关键点**：

- 序列号是严格递增的，幂等密钥是内容级的去重
- 序列号负责**顺序**，幂等密钥负责**去重**
- 两者结合提供完整的一致性保障

#### 乱序缓冲的上限

为了防止内存溢出，乱序缓冲必须有大小限制：

```
MAX_OUT_OF_ORDER_BUFFER_SIZE = 1000

if len(out_of_order_buffer) > MAX_OUT_OF_ORDER_BUFFER_SIZE:
    # 超过上限，说明网络严重异常或者设备故障
    log_error("out_of_order_buffer_overflow", session_id, seq, buffer_size)
    # 选项 1：主动关闭连接，迫使重连
    # 选项 2：丢弃最旧的缓冲消息（有风险）
```

### 9.4.5 一致性保障的多层设计

OpenClaw 通过多个层面确保分布式系统的一致性：

#### 层 1：协议级幂等（本节）

- 幂等密钥：识别重复请求
- 序列号：保证会话内顺序
- 乱序缓冲：收集乱序的消息，等待补齐

#### 层 2：存储级事务（上下文平面）

会话状态的每次更新都是原子的：

```
transaction {
    write session.history += new_event
    write session.state = computed_new_state
    write session.last_update_time = now()
}
```

如果事务中途失败（如磁盘满），整个更新回滚，不会产生中间状态。

#### 层 3：应用级约束（数据平面）

工具执行时的约束检查：

```
before_tool_execution:
    check tool_result not exists for this session:
        if exists: skip execution (already done)
    check no_conflicting_operations
    acquire_lock(session_id)

execute_tool:
    result = tool.run(args)

after_tool_execution:
    record_result_idempotent(result, idempotency_key)
    release_lock(session_id)
```

#### 层 4：审计与恢复（可观测性平面）

完整的操作日志，便于事后恢复：

```
audit_log:
    - timestamp: 1711100000
      event_type: TOOL_RESULT
      idempotency_key: ack_001
      session_id: sess_7f9e2a
      result: { success: true }
      action: "recorded"

    - timestamp: 1711100005
      event_type: TOOL_RESULT
      idempotency_key: ack_001
      session_id: sess_7f9e2a
      result: { success: true }
      action: "deduplicated (already recorded)"
```

### 9.4.6 实际案例：邮件发送的幂等保障

假设 Agent 需要发送一封邮件，网络中断导致响应丢失。让我们看看多层设计如何保证邮件只发一次：

```
[时间] [设备] [Gateway] [邮件服务]
  t=0  发送 TOOL_RESULT
       idempotency_key="ack_001"
       tool="send_email"
       result={...}
           ↓
  t=1      ↓ 收到并记录
      record idempotency_key="ack_001" → success
      发送 ACK
           ↓
  t=2      ↗ [网络中断，设备收不到 ACK]

  t=5  超时，重试
  发送 TOOL_RESULT (完全相同)
       idempotency_key="ack_001"
           ↓
  t=6      ↓ 查询缓存
      find idempotency_key="ack_001" → already recorded
      返回之前的结果 (success)
      不调用邮件服务（避免重复发送）
           ↓
  t=7  ↗ 收到 ACK，认为完成
```

在这个过程中：

1. **层 1（协议）**：识别 idempotency_key，触发去重逻辑
2. **层 2（存储）**：事务性地记录幂等密钥的结果
3. **层 3（应用）**：邮件服务只被调用一次
4. **层 4（审计）**：日志记录了去重的发生，便于事后验证

### 9.4.8 本节小结

分布式系统中，网络不可靠意味着重试是必须的。OpenClaw 通过多层一致性机制确保：

1. **幂等密钥**：识别重复请求，返回缓存结果而不是重复执行
2. **序列号**：保证会话内的事件按顺序处理
3. **乱序缓冲**：缓存乱序消息，等待补齐后顺序执行
4. **事务与审计**：存储与应用层的约束确保最终一致性

理解这些机制的关键是认识到：**至少一次的重试是可靠性的前提，幂等性是安全性的保证**。设计任何长连接、事件驱动系统时，都需要同时考虑这两个因素。
