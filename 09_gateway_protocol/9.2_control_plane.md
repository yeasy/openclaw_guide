## 9.2 控制平面职责与边界

Gateway 的控制平面是整个 OpenClaw 系统的"决策层"。与数据平面负责实际执行不同，控制平面只做**策略决策与状态协调**，不直接触及工具调用或模型推理。本节详解控制平面的五大职责边界。

### 9.2.1 五大职责及其边界

控制平面的职责可归纳为五类，它们按时间顺序贯穿整个请求生命周期：

#### 1. 连接认证与归属验证

**定义**：验证请求是否来自已授权的身份，并确定其归属（用户、团队、租户）。

**机制**：
- **认证**：通过密钥、OAuth、证书等确认"你是谁"
- **归属**：通过配对记录、会话令牌映射请求到具体的用户/Agent
- **隔离**：确保租户数据严格隔离，不存在越权访问

**例子**：用户 A 的 Telegram 消息到达 Gateway，控制平面检查该终端设备的配对记录，确认它属于用户 A 的账户，而不是被冒充。

**边界**：
- ✅ 做：验证身份、检查配对状态、映射到会话
- ❌ 不做：执行模型推理、调用工具、访问用户数据

#### 2. 路由与分发

**定义**：根据请求内容与策略规则，决定由哪个 Agent 来处理。

**机制**：
- **静态路由**：基于渠道、用户属性的确定性映射（如"Slack 消息由 Slack 助手处理"）
- **动态路由**：基于消息内容、优先级、负载的智能分发（如"紧急问题转向 escalation_agent"）
- **策略路由**：支持条件规则、优先级队列、负载均衡

**例子**：一条包含"紧急"的消息到达，控制平面的路由规则检查匹配到 `urgency: critical` 的目标 Agent，将请求转向专门的 escalation_agent。

**边界**：
- ✅ 做：匹配规则、选择目标 Agent、决定优先级队列
- ❌ 不做：修改消息内容、决定 Agent 的响应方式

#### 3. 会话与状态管理

**定义**：为每个用户维护独立的会话，追踪上下文与执行状态。

**机制**：
- **会话生命周期**：创建→活跃→冻结/超时→过期
- **状态持久化**：每一步决策都持久化到存储（便于恢复与审计）
- **并发控制**：同一会话的请求进行排序与互斥，避免竞态

**例子**：用户 A 的第 10 条消息到达时，控制平面查找前 9 条消息的历史记录、工具结果与系统状态，确保 Agent Loop 可以基于完整上下文推理。

**边界**：
- ✅ 做：管理会话生命周期、持久化状态、控制并发
- ❌ 不做：决定上下文压缩策略（由上下文平面负责）

#### 4. 策略执行与约束检查

**定义**：在执行前检查策略约束，确保请求符合访问控制、工具限制、成本预算等要求。

**机制**：
- **工具黑白名单**：此 Agent 允许/禁用哪些工具
- **成本预算**：当前会话是否超出 Token 预算、速率限制
- **用户同意机制**：高风险操作是否获得用户显式同意
- **HITL 门控**：某些步骤是否需要人工审批前执行

**例子**：请求尝试调用"删除文件"工具，但该 Agent 的配置中"删除文件"在黑名单中，控制平面拒绝。

**边界**：
- ✅ 做：检查黑白名单、速率限制、成本预算、同意状态、HITL 门控
- ❌ 不做：执行工具本身（由数据平面做）

#### 5. 故障恢复与熔断策略

**定义**：当运行时遇到故障（超时、限流、模型失败等），决定如何应对。

**机制**：
- **超时与重试**：配置单步超时与重试次数
- **回退链路**：触发高成本模型失败时，切换到低成本模型
- **熔断**：连续失败超过阈值时，自动拒绝新请求，避免级联故障
- **降级策略**：当核心依赖不可用时，是否返回降级响应还是直接失败

**例子**：模型 A 连续返回超时，控制平面触发回退规则，切换到模型 B。如果模型 B 也失败，则按配置决定是否继续重试或降级返回预设回答。

**边界**：
- ✅ 做：决定重试策略、触发回退链路、执行熔断
- ❌ 不做：改变模型推理过程、自行执行工具（由 Agent Loop 负责）

### 9.2.2 控制平面的设计模式：分层审批

在生产环境中，控制平面的决策往往需要多层审核，以确保安全性与可追溯性。典型的分层审批模式包括：

```
请求到达
    ↓
[第一层] 认证与配对检查 ← 信任平面：密钥有效？配对有效？
    ↓
[第二层] 路由与优先级检查 ← 控制平面：这条消息应转向哪个 Agent？
    ↓
[第三层] 策略约束检查 ← 信任平面：工具黑白名单、用户同意、HITL 门控
    ↓
[第四层] 资源预算检查 ← 可观测性平面：是否超出 Token 预算、速率限制？
    ↓
[批准] → 转向 Agent Loop 执行
```

每一层都有**明确的决策规则与失败路径**，任何一层的拒绝都意味着请求被终止并返回相应的拒绝原因（供审计与调试）。

### 9.2.3 控制平面与数据平面的严格隔离

为了提高系统的可维护性与安全性，控制平面与数据平面必须保持严格隔离：

**控制平面（只做决策）**：
- 检查请求是否被允许
- 决定路由与优先级
- 触发重试/回退
- 暂停 HITL 门控

**数据平面（只做执行）**：
- 执行工具调用
- 运行模型推理
- 处理流式输出
- 管理沙箱隔离

**通信方式**：
- 控制平面 → 数据平面：发出"执行 X 工具"、"调用 Y 模型"的指令
- 数据平面 → 控制平面：上报"X 工具执行成功"、"Y 模型返回超时"的结果
- **禁止反向控制**：数据平面不能自主决定重试/回退，必须向控制平面报告，由控制平面决策

### 9.2.4 实际配置示例

在 OpenClaw 的 `openclaw.json` 中，控制平面的配置通常如下（简化示例）：

```json
{
  "agents": {
    "default": {
      "model": "gpt-4o",
      "fallbacks": ["gpt-4-turbo", "gpt-3.5-turbo"],
      "routing": {
        "rules": [
          {
            "match": { "field": "priority", "equals": "critical" },
            "target": "escalation_agent"
          },
          {
            "match": { "field": "channel", "equals": "ops" },
            "target": "ops_agent"
          }
        ],
        "fallback": "default_agent"
      }
    }
  },
  "policies": {
    "toolBlacklist": ["delete_file", "drop_database"],
    "toolWhitelist": ["search", "read_file", "send_email"],
    "costBudget": {
      "perSession": 5000,
      "perDay": 100000
    },
    "rateLimit": {
      "requestsPerMinute": 100
    },
    "hitl": {
      "requiresApproval": ["execute_migration", "send_email_bulk"]
    }
  },
  "resilience": {
    "timeout": {
      "model": 30,
      "tool": 60
    },
    "retry": {
      "maxAttempts": 3,
      "backoffMultiplier": 2
    },
    "circuitBreaker": {
      "failureThreshold": 5,
      "resetTimeout": 300
    }
  }
}
```

**解读**：
- `routing`：匹配规则确定目标 Agent（第 2 层职责）
- `policies`：工具限制、预算、速率限制（第 3、4 层职责）
- `resilience`：超时、重试、熔断配置（第 5 层职责）

### 9.2.5 常见陷阱与最佳实践

#### 陷阱 1：控制平面做了数据平面的工作

❌ **错误做法**：控制平面在路由时修改消息内容、注入提示词修饰语。

✅ **正确做法**：控制平面只决定"转向 Agent A"，由 Agent Loop 负责提示词装配与修饰。

#### 陷阱 2：策略检查不一致

❌ **错误做法**：工具黑名单在两个地方维护（配置文件 + 代码硬编码），导致不一致。

✅ **正确做法**：策略集中在一处（如 `openclaw.json`），所有检查都从同一源读取。

#### 陷阱 3：故障恢复决策不透明

❌ **错误做法**：触发回退后，日志中看不到"为什么切换模型"的决策链。

✅ **正确做法**：每一次回退都记录触发规则、命中条件、新目标，便于审计与复盘。

#### 最佳实践

1. **配置即代码**：将所有控制平面的策略配置化，避免硬编码。
2. **决策透明化**：每个决策都在日志中留下"为什么"的痕迹。
3. **定期演练**：进行故障注入测试，验证回退、重试、熔断是否如预期工作。
4. **分离关注点**：控制平面关注"准许/拒绝"，数据平面关注"怎么执行"。
