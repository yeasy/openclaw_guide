## 1.2 适用场景与竞品对比

### 1.2.1 适合做什么，不适合做什么

| 适合做什么 | 不适合做什么 |
|---|---|
| **跨渠道统一入口**：通过 Telegram、WhatsApp、飞书等多个渠道统一接入，在习惯的聊天工具里直接调度 AI | **纯对话问答**：如果只需一个聊天助手回答日常问题，直接用 ChatGPT 或 Claude，不需要架设网关 |
| **本地执行与数据主权**：数据和执行过程留在自有服务器上，适合内网运维、敏感文档处理等对隐私有要求的场景 | **零运维的托管服务**：OpenClaw 需要自行部署和维护。没有运维能力或意愿的团队，托管 SaaS 更省心 |
| **带权限防护的内部工具链**：连接内部知识库和基础设施，允许授权用户通过自然语言触发操作，同时用网关拦截越权指令 | **大规模企业级多租户**：当前版本偏向个人或小团队。需要完整 RBAC、多租户隔离和 SLA 保证的场景，建议评估专业企业平台 |
| **长时间运行的自动化任务**：结合 Cron 定时任务和 Webhook 触发器，让智能体常驻运行，主动巡检、拉取数据、定期报告 | **对延迟极度敏感的实时系统**：每次工具调用都涉及 LLM 推理，响应通常在秒级。毫秒级响应场景（高频交易、实时游戏）不适合 |
| **复杂多步工具编排**：内置文件系统、Shell、HTTP、浏览器、数据库等数十种工具，支持智能体在多步推理中灵活调用 | **逻辑固定的简单自动化**：如果任务逻辑清晰且固定（如"收到邮件→转发到 Slack"），Zapier 或 n8n 更高效，不需要引入 LLM 的推理开销 |

### 1.2.2 与主流工具的对比

| 工具 | 核心定位 | 与 OpenClaw 的关键区别 |
|---|---|---|
| **ChatGPT / Claude（对话客户端）** | 一问一答的智能问答 | 对话结束后状态消散，没有工具执行能力，无法主动触发任务。**如果只需聊天，用它们就够了** |
| **Cursor** | 针对编程场景深度优化的 AI 助手 | 专才系统，开箱即用。OpenClaw 是**构建各类 AI 助手的通用底座**，适合非固定的业务场景 |
| **Zapier / n8n** | 基于硬编码规则的自动化工作流 | 规则固定，遇到格式偏差容易瘫痪。OpenClaw 基于自然语言推理，**能处理模糊输入和意外报错** |
| **Dify / Coze** | 低代码 AI 应用构建平台，云端托管 | 开箱即用但依赖云端。OpenClaw 提供**完全私有化部署**、细粒度网关权限、本地工具执行能力 |
| **LangChain / AutoGen** | 构建智能体的开发框架 | 需要大量编写代码组装。OpenClaw 偏向**开箱即用的运行时**，内置网关、多渠道、记忆管理，适合个人/小团队快速落地 |

> **选型建议**：如需个人学习或日常问答 → ChatGPT/Claude；如需编程助手 → Cursor；如需在**自有服务器上构建可连接真实工具的智能体** → OpenClaw。

### 附：Token 成本要有清醒认识

OpenClaw 的每次交互都会消耗模型 API 的 Token。与普通对话不同，一次任务可能触发 2–5 轮推理，每轮都会重新发送系统提示、工具定义和上下文历史。**月度账单可能比你预期的高出数倍。**

几个控制成本的基本原则：
- 只挂载当前场景需要的工具，减少每轮的工具定义开销
- 合理配置会话重置，防止历史消息无限膨胀
- 简单任务用轻量模型（如 Haiku），复杂推理再用 Sonnet 或 Opus
- **只在需要自然语言理解、模糊决策或多步推理的环节使用智能体**，其余部分用确定性代码实现

详细的 Token 成本分析见[第十四章 性能与成本优化](../14_performance_cost/README.md)。
